## Цель
Это задание — логическое продолжение ДЗ №2 про single-leader репликацию.

В single-leader системе запись идёт через одну основную точку.
Это проще для реализации, но ограничивает систему: все write-запросы сходятся в один узел,
а удалённым клиентам неудобно писать только в него.

В этой домашке нужно реализовать **multi-leader режим**: в кластере есть **несколько узлов, принимающих запись**,
а изменения затем распространяются по системе. Такой режим даёт несколько точек записи, но создаёт новую проблему:
один и тот же ключ может почти одновременно измениться на разных leader-узлах.

Значит, в системе появляются **конфликты версий**,
а ваша задача — разрешать их **детерминированно** и в итоге приводить все узлы к одному состоянию.

В рамках этого задания кластер состоит из **10 узлов**, из которых **4 — `leader`**, а **6 — `follower`**.

## 1. Что нужно переиспользовать из ДЗ №2
Новое решение должно расширять существующий проект, а не быть отдельной реализацией.

Нужно переиспользовать:
- процессы `Node`;
- CLI / Cluster Controller;
- TCP-сокеты;
- формат сообщений `JSON Lines`;
- membership-команды: `addNode`, `removeNode`, `listNodes`;
- рассылку `CLUSTER_UPDATE`;
- `requestId`, `clientId`, `operationId`;
- in-memory дедупликацию операций с TTL;
- команду `setReplicationDelayMs`;
- single-leader режим из ДЗ №2.

CLI должен оставаться единой точкой входа в систему.

## 2. Что требуется и что не требуется
### Требуется
- режим `multi`;
- кластер из 6 узлов: `2 leader + 4 follower`;
- запись только через `leader`-узлы;
- репликация между узлами;
- eventual consistency;
- конфликты версий и их детерминированное разрешение;
- несколько топологий распространения обновлений;
- демонстрационные сценарии;
- бенчмарки и краткий отчёт.

### Не требуется
- выбор лидера;
- хранение на диске.

## 3. Архитектура и режимы работы
В системе должны быть минимум два типа процессов:
1. **Node** — узел key-value хранилища;
2. **CLI / Cluster Controller** — консольное приложение для управления кластером и отправки запросов.

CLI должен уметь:
- управлять составом кластера;
- переключать режимы работы;
- задавать топологию репликации;
- отправлять пользовательские команды;
- запускать демонстрационные сценарии и бенчмарки.

Все изменения membership и конфигурации должны доходить до узлов по сети через `CLUSTER_UPDATE`.

Поддерживаются два режима:
- `single` — поведение из ДЗ №2;
- `multi` — новый режим этого задания.

CLI должен поддерживать `setMode single|multi`.

### `mode=single`
Поведение остаётся как в ДЗ №2.

### `mode=multi`
В `multi`-режиме:
- в кластере есть **ровно 4 leader-узла**, принимающих клиентские записи;
- `follower`-узлы клиентские записи не принимают;
- изменения, принятые разными `leader`, должны реплицироваться по кластеру;
- система должна приходить к eventual consistency.

`setLeader` можно оставить ради совместимости с ДЗ №2, но в `mode=multi` вместо одного лидера используется **заранее заданный набор leader-узлов**, приходящий через конфигурацию кластера.

## 4. Роли узлов и обработка `PUT`
У каждого узла в `mode=multi` есть роль: `leader` или `follower`.

Для этого задания достаточно считать, что в кластере из 10 узлов:
- 4 узла имеют роль `leader`;
- 6 узлов имеют роль `follower`.

Роли должны распространяться вместе с конфигурацией кластера.

### Правило записи
- `leader` принимает `PUT`;
- `follower` не принимает клиентский `PUT`.

Если клиент отправил `PUT` на `follower`, узел должен вернуть ошибку и подсказать, к какому leader-узлу нужно идти.

Минимальные требования к ответу:
- `status: ERROR`
- `errorCode: NOT_LEADER_FOR_WRITE`
- `leaderNodeId: <uuid leader-узла>`

Дополнительно можно вернуть адрес leader-узла, но это не обязательно.

## 5. Модель данных
В multi-leader режиме недостаточно хранить просто значение. Нужно хранить значение вместе с версией.

Минимальный вариант:
```text
Map<String, VersionedValue>

VersionedValue {
  value: String
  version: {
    lamport: long
    nodeId: String
  }
}
```

Где:
- `lamport` — версия по Lamport Clock;
- `nodeId` — идентификатор узла, создавшего версию.

`DELETE` не обязателен.

## 6. Версионирование: Lamport Clock
Для multi-leader режима нужно реализовать **Lamport Clock**.

У каждого узла есть локальный счётчик:
```text
lamportCounter: long
```

### Базовые правила
При локальном `PUT` на leader-узле:
```text
L := L + 1
version := (L, currentNodeId)
```

При получении репликационной операции с версией `(remoteLamport, remoteNodeId)`:
```text
L := max(L, remoteLamport) + 1
```

Важно: при применении репликации в хранилище для ключа должна сохраняться **именно пришедшая версия**, а не новая локальная.

Lamport Clock не восстанавливает физическое время.
Здесь он нужен как простой способ выдавать операциям упорядочиваемые версии,
чтобы затем одинаково сравнивать их на всех узлах.

### Почему значения `lamport` могут совпасть
Одинаковые значения `lamport` — нормальный сценарий для multi-leader.

Пример:
- у leader `A` локальный счётчик равен `5`;
- у leader `B` локальный счётчик тоже равен `5`;
- почти одновременно оба принимают `PUT` одного и того же ключа;
- оба увеличивают счётчик до `6`.

В результате появляются две версии:
```text
(6, nodeA)
(6, nodeB)
```

Значит, одного `lamport` недостаточно: при равенстве нужно дополнительное правило сравнения.

## 7. Разрешение конфликтов
Нужно реализовать **LWW (Last Write Wins)** на основе версии:
```text
version = (lamport, nodeId)
```

Сравнение версий должно быть единым на всех узлах:
- версия новее, если у неё больше `lamport`;
- если `lamport` одинаковый — используется tie-break по `nodeId`.

Например, можно сравнивать пары лексикографически:
```text
(l1, n1) > (l2, n2)
если
l1 > l2
или
l1 == l2 && n1 > n2
```

При применении входящей версии:
- если ключа ещё нет — сохранить значение;
- если `incoming.version > local.version` — заменить локальное значение;
- иначе — проигнорировать входящую версию.

### Что должно получиться
Если прекратить новые записи и дать репликации завершиться,
все узлы должны прийти к одному и тому же значению для каждого ключа.

## 8. Сетевой протокол и типы сообщений
### Транспорт и формат
- TCP-сокеты;
- допустимо использовать один порт на узел;
- тип сообщения определяется полем `type`;
- рекомендуемый формат — **JSON Lines**: один JSON-объект на строку.

### Клиентские запросы
Минимум должны поддерживаться:
- `CLIENT_PUT`
- `CLIENT_GET`
- `CLIENT_DUMP`

Общие поля:
- `requestId`
- `clientId`
- `type`

Для `CLIENT_PUT`: `key`, `value`  
Для `CLIENT_GET`: `key`

### Репликационные сообщения
#### `REPL_PUT`
Обязательные поля:
- `type: REPL_PUT`
- `operationId`
- `originNodeId`
- `sourceNodeId`
- `key`
- `value`
- `version: { lamport, nodeId }`

#### `REPL_ACK`
Нужен для логов и метрик.

Обязательные поля:
- `type: REPL_ACK`
- `operationId`
- `fromNodeId`

### Ошибки
В ответе должны быть:
- `requestId`
- `status: OK | ERROR`

Если `status=ERROR`, должен быть `errorCode`.

Минимально рекомендуемые коды:
- `BAD_REQUEST`
- `UNKNOWN_NODE`
- `INVALID_MODE`
- `INVALID_TOPOLOGY`
- `INVALID_CONFIG`
- `TIMEOUT`
- `NOT_LEADER_FOR_WRITE`

## 9. Идемпотентность и общая механика репликации
В multi-leader режиме дубликаты сообщений допустимы, особенно в `ring` и при повторной отправке.
Нужно реализовать in-memory дедупликацию по `operationId`.

Минимальные требования:
- у каждого узла есть `seenOperationIds` с TTL;
- если `operationId` уже встречался, операция не применяется повторно и не форвардится дальше;
- допустимо повторно вернуть `ACK`;
- должна быть очистка старых записей.

Рекомендуемый TTL: около 5 минут.

### Как обрабатывается локальный `PUT` в `mode=multi`
1. запрос приходит на узел;
2. если это `follower`, он возвращает `NOT_LEADER_FOR_WRITE` и `leaderNodeId`;
3. если это `leader`, он:
   - валидирует запрос;
   - увеличивает `lamportCounter`;
   - создаёт `operationId`;
   - создаёт версию `(lamportCounter, currentNodeId)`;
   - применяет значение локально;
   - распространяет `REPL_PUT` по кластеру согласно текущей топологии.

В рамках этого задания достаточно **best effort delivery**.

## 10. Топологии репликации
В этом задании топология — одна из центральных идей.
Недостаточно просто "как-нибудь разослать" обновление: важно, **по какому графу оно идёт**,
сколько делает хопов, где возникают узкие места и как быстро все узлы приходят к одному состоянию.

CLI должен поддерживать:
- `setTopology mesh|ring|star`
- `setStarCenter <nodeId>`

Топология определяет **правило форвардинга** репликационных сообщений.

### Общие требования
- одна и та же топология должна одинаково интерпретироваться на всех узлах;
- после `CLUSTER_UPDATE` узлы должны использовать актуальную топологию;
- при форвардинге нужно сохранять `operationId`, `originNodeId` и версию;
- дедупликация должна предотвращать бесконечное хождение одного и того же обновления по графу;
- в `mode=multi` топология должна реально влиять на путь распространения сообщений и на время схождения.

### 10.1. Mesh
В `mesh` узел, впервые получивший новую операцию, распространяет её максимально широко.
Это топология с высокой связностью: она обычно даёт самое быстрое схождение, но и самый большой сетевой трафик.

Что требуется:
- leader после локального `PUT` рассылает `REPL_PUT` всем остальным узлам;
- дальше можно выбрать один из двух вариантов:
   - либо остальные узлы не форвардят сообщение, если лидер уже разослал его всем напрямую;
   - либо узел, впервые увидевший операцию, форвардит её всем остальным, кроме отправителя;
- выбранную схему нужно коротко зафиксировать в `README.md`.

Что должно быть продемонстрировано:
- обновление распространяется по наиболее "широкому" графу;
- `mesh` даёт быстрое схождение;
- за более быстрое схождение система платит большим числом отправок.

### 10.2. Ring
В `ring` узлы образуют логическое кольцо.
Каждый узел знает своего следующего соседа и пересылает обновление только ему.
Эта топология экономнее по числу прямых отправок, но обновление идёт через несколько хопов, поэтому схождение обычно медленнее.

Порядок кольца должен определяться детерминированно, например сортировкой `nodeId`.
Нельзя полагаться на случайный порядок добавления узлов.

Что требуется:
- по текущему membership должен строиться единый порядок узлов в кольце;
- для каждого узла должен однозначно определяться `next`;
- leader после локального `PUT` отправляет `REPL_PUT` следующему узлу в кольце;
- каждый следующий узел, если видит `operationId` впервые, применяет операцию и пересылает её дальше;
- благодаря дедупликации операция не должна бесконечно ходить по кольцу.

Что должно быть продемонстрировано:
- путь обновления хорошо виден по логам и отладочным командам;
- время схождения выше, чем у `mesh`;
- без корректного `removeNode` кольцо может ломаться и переставать доставлять обновления дальше.

### 10.3. Star
В `star` есть центральный узел (`star center`), через который проходят репликационные сообщения.
Такая топология обычно проще для reasoning и отладки, но создаёт явное узкое место в центре.

CLI должен поддерживать `setStarCenter <nodeId>`.

Что требуется:
- если изменение возникло на нецентральном узле, оно должно идти через центр;
- центр должен пересылать обновление остальным релевантным узлам;
- если изменение возникло на центре, он сам рассылает его остальным;
- смена `star center` через CLI должна реально менять путь распространения сообщений.

Что должно быть продемонстрировано:
- в штатном режиме `star` работает корректно;
- центральный узел становится bottleneck;
- при остановке центра поведение заметно ухудшается или нарушается вовсе.

### 10.4. Что нужно показать в итоге
Для всех трёх топологий должно быть возможно показать:
- что обновления действительно идут по разным путям;
- что `mesh`, `ring` и `star` различаются по скорости схождения;
- что влияние топологии видно через `getAll`, `clusterDump`, логи и бенчмарки.

## 12. CLI-команды
### Управление кластером
CLI должен поддерживать следующие команды:
- `addNode <nodeId> <host> <port>` — добавить узел в membership;
- `removeNode <nodeId>` — удалить узел из membership;
- `listNodes` — показать текущий состав кластера;
- `setMode single|multi` — переключить режим работы;
- `setLeader <nodeId>` — оставить ради совместимости с ДЗ №2;
- `setTopology mesh|ring|star` — выбрать топологию репликации;
- `setStarCenter <nodeId>` — задать центральный узел для `star`;
- `setReplicationDelayMs <min> <max>` — задать искусственную задержку репликации.

После изменения membership или конфигурации CLI должен разослать актуальный `CLUSTER_UPDATE` всем живым узлам.

### Пользовательские команды
- `put <key> <value> [--target <nodeId>] [--client <clientId>]` — записать значение на конкретный узел;
- `get <key> [--target <nodeId>] [--client <clientId>]` — прочитать значение с конкретного узла;
- `dump [--target <nodeId>]` — вывести полное локальное состояние выбранного узла.

### Отладочные команды
Нужно реализовать **обе** команды:
- `getAll <key>` — сделать `GET` на все узлы и вывести состояние ключа на каждом узле в формате `value + version`;
- `clusterDump` — сделать `dump` на все узлы и вывести их полное состояние.

Обе команды нужны для демонстрации расхождений, lag и последующего схождения.

## 14. Benchmarks и отчёт
Задача бенчмарков в этой домашке — не просто получить красивые цифры, а показать, **как архитектурные решения влияют на поведение системы**. Нас интересуют не только throughput и latency, но и то, как разные топологии сходятся, где возникает bottleneck и что меняется при росте конфликтности.

### 14.1. Что измерять
Минимум:
- throughput (`ops/sec`);
- latency: `avg`, `p50`, `p75`, `p95`, `p99`;
- сравнение разных топологий;
- хотя бы одна метрика времени схождения (`convergence time`).

Результаты нужно сохранить в `benchmarks/results.csv` или `benchmarks/results.json`.

### 14.2. Обязательные прогоны
Кластер: **10 узлов** (4 `leader` + 6 `follower`). Для throughput-бенчмарков задержка репликации должна быть `0`.

#### Часть A — сравнение топологий
Нужно выполнить **6 прогонов**:
- `mode=multi`
- `topology = mesh, ring, star`
- `putRatio = 0.8` и `putRatio = 0.2`
- `threads = 16`
- `totalOps >= 200_000`
- `keySpace = 10_000`

Это нужно, чтобы сравнить поведение трёх топологий при write-heavy и read-heavy нагрузке.

#### Часть B — влияние конфликтности
Нужно выполнить **2 прогона**:
- `mode=multi`
- `topology=mesh`
- `putRatio=1.0`
- `threads=32`
- `totalOps >= 200_000`

Прогоны:
1. `keySpace = 5`
2. `keySpace = 10_000`

Идея проста: маленький `keySpace` создаёт больше конкуренции за одни и те же ключи и обычно даёт больше конфликтов, чем большой `keySpace`.

#### Часть C — сравнение single-leader и multi-leader
Нужно выполнить минимум **4 прогона**:
1. `mode=single`, `replication=async`, `RF=1`
2. `mode=single`, `replication=sync`, `RF=3`
3. `mode=multi`, `topology=mesh`
4. `mode=multi`, `topology=star`
