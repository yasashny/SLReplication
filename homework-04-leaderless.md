## Цель работы
Это задание — продолжение ДЗ №2 и ДЗ №3.
Нужно реализовать **leaderless-режим** для учебного in-memory key-value хранилища:
- в системе **нет фиксированного лидера**;
- **любой узел** может принять клиентский запрос;
- узел, принявший запрос, становится **координатором этого запроса**;
- запись и чтение выполняются по правилам **quorum**;
- система должна поддерживать **strict quorum**, **sloppy quorum только для записи**, **hinted handoff**, **async read-repair** и **anti-entropy через Merkle Tree**.
Дополнительное обязательное требование: после полной потери данных на одной из home-реплик узел должен восстановить своё состояние **из остальных home-реплик кластера** с помощью **cluster-wide anti-entropy**, использующей **Merkle Tree**.
Все данные во всех режимах хранятся **только in-memory**.

## 3. Архитектура и фиксированная модель кластера
В системе должны быть минимум два типа процессов:
1. **Node** — узел key-value хранилища.
2. **CLI / Cluster Controller** — консольное приложение для управления кластером, отправки клиентских запросов, запуска hinted handoff, anti-entropy и benchmark-сценариев.

### 3.1. Фиксированный размер кластера
- всего **7 узлов**;
- `5` узлов — **home replicas**;
- `2` узла — **spare nodes**.

### 3.2. Упрощение вместо consistent hashing
В этом ДЗ **не используется consistent hashing**. Для простоты считается, что:
- **для всех ключей** используется один и тот же набор из **5 home-реплик**;
- spare-узлы не являются home-репликами ни для какого ключа;
- spare-узлы используются только как временное место хранения hints при sloppy quorum.

### 3.3. Координатор запроса
В режиме `leaderless` любой узел кластера может принять `PUT`, `GET` и отладочные команды.
Узел, принявший конкретный запрос клиента, становится **координатором этого запроса**.
Координатор отвечает за рассылку write/read запросов home-репликам, подсчёт ACK, выбор итогового значения при чтении, запуск async read-repair, организацию sloppy quorum при записи, запуск hinted handoff и anti-entropy по команде CLI.
Координатор **не является лидером** и не задаёт глобальный порядок операций в кластере.

## 4. Модель данных и версионирование
### 4.1. Основное хранилище
На каждой home-реплике должно быть in-memory хранилище:
```text
Map<String, VersionedValue>
```
Минимальная обязательная модель:
```text
VersionedValue {
  value: String
  version: { lamport: long, nodeId: String }
  operationId: String
}
```

### 4.2. Хранилище hints
Hints должны храниться **отдельно** от основного store.
```text
HintRecord {
  key: String
  value: String
  version: { lamport: long, nodeId: String }
  operationId: String
  intendedHomeNodeId: String
}
```

### 4.3. Lamport version
Для каждой локальной записи координатор должен формировать версию:
```text
version = (lamport, nodeId)
```
Правила:
1. У каждого узла есть локальный `lamportCounter`.
2. При локальном `PUT`, который координирует узел:
```text
lamportCounter := lamportCounter + 1
version := (lamportCounter, currentNodeId)
```
3. При получении записи с удалённой версией `(remoteLamport, remoteNodeId)` узел должен обновить локальный счётчик так, чтобы он не отставал от увиденной версии.
4. При применении входящей записи в store сохраняется **входящая версия**, а не новая локальная.

### 4.4. Сравнение версий
Запись новее, если у неё больше `lamport`; при равенстве `lamport` победитель определяется по `nodeId`.
Лексикографическое сравнение:
```text
(l1, n1) > (l2, n2)
если
l1 > l2
или
l1 == l2 && n1 > n2
```
Это правило должно использоваться при чтении, read-repair, hinted handoff и anti-entropy recovery.

## 5. Кворумная модель
Число home-реплик фиксировано: **`N = 5`**.
CLI должен позволять настраивать `W` и `R`.
Ограничения:
- `1 <= W <= 5`
- `1 <= R <= 5`
- **обязательно:** `W + R > 5`
Если конфигурация не удовлетворяет этим условиям, команда должна завершаться ошибкой `INVALID_CONFIG`.
`W` и `R` настраиваются только относительно **5 home-реплик**. Spare-узлы не участвуют в обычном read quorum и могут участвовать только в **write-path при sloppy quorum**.

## 6. Сетевой протокол
### 6.1. Транспорт и формат
- TCP-сокеты;
- допустимо использовать один порт на узел;
- формат сообщений — **JSON Lines** (`1 JSON = 1 строка`).

### 6.2. Обязательные типы сообщений
**Клиентские / управляющие:** `CLIENT_PUT`, `CLIENT_GET`, `CLIENT_DUMP`, `CLIENT_DUMP_HINTS`, `CLUSTER_UPDATE`.

**Внутрикластерные для leaderless write/read:** `REPL_WRITE`, `REPL_WRITE_ACK`, `READ_QUERY`, `READ_RESPONSE`.

**Hinted handoff:** `HINTED_HANDOFF_TRANSFER`, `HINTED_HANDOFF_ACK`.

**Merkle / anti-entropy:** `MERKLE_ROOT_REQUEST`, `MERKLE_ROOT_RESPONSE`, `MERKLE_DIFF_REQUEST`, `MERKLE_DIFF_RESPONSE`, `MERKLE_RECORDS_TRANSFER`, `MERKLE_RECORDS_ACK`.

### 6.3. Минимальные обязательные поля
У каждого клиентского запроса должны быть `requestId`, `clientId`, `type`.
У каждой операции записи должны быть `operationId`, `key`, `value`, `version`.
Для hinted handoff дополнительно нужен `intendedHomeNodeId`.

### 6.4. Минимальный набор ошибок
Достаточно поддержать `BAD_REQUEST`, `INVALID_CONFIG`, `NOT_ENOUGH_REPLICAS`, `TIMEOUT`, `NODE_UNAVAILABLE`.

## 7. Дедупликация и идемпотентность
Любая внутренняя операция должна применяться **не более одного раза**.
Минимальные требования:
- in-memory `seenOperationIds` с TTL;
- дубликаты не должны повторно применяться;
- допустимо повторно отправлять ACK на уже обработанную операцию;
- должна быть периодическая очистка старых `operationId`.
Рекомендуемый TTL: около **5 минут**.
Дедупликация обязательна для `REPL_WRITE`, `HINTED_HANDOFF_TRANSFER`, `MERKLE_RECORDS_TRANSFER`.

## 9. Write path: strict quorum
При `PUT` координатор должен:
1. провалидировать запрос;
2. увеличить локальный `lamportCounter`;
3. создать `operationId`;
4. создать `version = (lamport, coordinatorNodeId)`;
5. разослать `REPL_WRITE` на все **5 home-реплик**;
6. ждать подтверждений согласно текущему режиму write quorum.

В режиме `strict` запись считается успешной, если координатор получил подтверждения минимум от **W home-реплик**.
Если `W` подтверждений не набрано до таймаута, клиент должен получить `status = ERROR` и `errorCode = NOT_ENOUGH_REPLICAS` или `TIMEOUT`.
Подтверждения от spare-узлов в strict-режиме **не учитываются**.

## 10. Sloppy quorum и hinted handoff

В этом ДЗ sloppy quorum поддерживается **только для записи**. Чтения всегда выполняются только по **home-репликам**.

Если в режиме `sloppy` часть home-реплик недоступна, координатор может использовать spare-узлы, чтобы всё равно набрать `W` подтверждений.

Пример: home-реплики `H1 H2 H3 H4 H5`, spare-узлы `S1 S2`, `W = 4`, `H4` недоступна. Тогда координатор может получить ACK от `H1`, `H2`, `H3`, `S1` и считать запись успешной.
Spare-узел не должен записывать такую операцию в основной home-store как обычную реплику. Он должен сохранять её как **HintRecord** с полным значением, версией, `operationId` и `intendedHomeNodeId`.
После восстановления недоступной home-реплики узел, хранящий hint, должен попытаться переслать её через `HINTED_HANDOFF_TRANSFER`. После успешной доставки hint можно удалить.
Важные правила:
- hint не является обычной home-репликой;
- запись, сохранённая как hint, **не участвует в обычном чтении**;
- hints не включаются в Merkle Tree.

## 11. Read path и async read-repair

При `GET` координатор должен:
1. отправить `READ_QUERY` всем **5 home-репликам**;
2. дождаться минимум **R успешных ответов**;
3. среди полученных ответов выбрать запись с максимальной версией;
4. вернуть её клиенту.

Если собрать `R` ответов не удалось — вернуть ошибку.
Если при чтении координатор увидел, что часть ответивших home-реплик содержит более старое значение, он должен запланировать **асинхронное исправление** этих реплик.
Требования:
- read-repair в этом ДЗ должен быть **только async**;
- клиентский `GET` не должен ждать завершения repair;
- stale home-репликам должен быть отправлен более новый `VersionedValue`.
Read-repair не чинит hints, не заменяет anti-entropy и не гарантирует, что редко читаемые ключи автоматически починятся.

## 12. Merkle Tree и anti-entropy

Merkle Tree строится только по **основному store home-реплики**. В дерево **не включаются** hints, `seenOperationIds`, служебные очереди и любые временные структуры.
Чтобы решение было однозначным и проверяемым, требуется следующая схема.

**Фиксированное число бакетов:** `B = 16`.

**Разбиение ключей на бакеты:**
```text
bucket = abs(hash(key)) mod 16
```

**Хэш листа:**
1. взять все записи бакета;
2. отсортировать по `key`;
3. сериализовать запись как строку:
```text
key|value|lamport|nodeId|operationId
```
4. вычислить хэш бакета как:
```text
SHA-256(concatenation_of_sorted_records)
```
Если бакет пустой:
```text
SHA-256("")
```

**Внутренние узлы дерева:**
```text
SHA-256(leftHash + rightHash)
```
Корень дерева строится снизу вверх из 16 leaf-бакетов.
Команда `showMerkleRoot <nodeId>` должна выводить текущий root hash выбранного узла. Это обязательная часть задания.

---

## 13. Cluster-wide anti-entropy
В этом ДЗ anti-entropy **не запускается в фоне**. Она должна запускаться **явной командой**:
```text
runAntiEntropyCluster
```

Команда должна выполнять **один anti-entropy-pass по всем живым home-репликам**.
Минимально допустимая семантика:
1. собрать Merkle root всех живых home-реплик;
2. найти пары home-реплик, у которых root различается;
3. для каждой такой пары выполнить Merkle-based diff;
4. локализовать различающиеся бакеты;
5. обменяться только записями из различающихся бакетов;
6. для каждого ключа оставить запись с более новой версией;
7. завершить pass с обновлёнными root.
Ограничения:
- `runAntiEntropyCluster` должна работать **по home-репликам**, а не копировать данные на spare-узлы как обычные реплики;

## 14. Восстановление ноды после потери данных
Команда

```text
wipeNodeData <nodeId>
```
должна очистить основной store узла, hints узла и временные in-memory данные, связанные с содержимым узла. Это имитирует падение узла с полной потерей in-memory состояния.
Если wiped-узел является home-репликой, то после запуска
```text
runAntiEntropyCluster
```
он должен восстановить актуальные данные из остальных home-реплик.
Recovery должен происходить через **Merkle-based anti-entropy**, а не через слепую полную копию всей `Map` с одной ноды на другую.
Именно для этого в обязательной части присутствуют `showMerkleRoot`, Merkle diff и передача только записей из отличающихся бакетов.

### 19.2. Качество кода — 2 балла

**1. Архитектура и проектирование — 1.0 балл**
Оцениваются:

* разумное разделение ответственности между модулями;
* низкая связность и понятные интерфейсы между слоями;
* ясные модели сообщений и доменных сущностей;
* расширяемая структура проекта без хаотичной логики и “больших классов”.

Плюсом считается уместное применение проектных подходов и паттернов там, где они действительно упрощают код и делают его поддерживаемым.

**2. Надёжность и читаемость — 1.0 балл**
Оцениваются:

* потокобезопасность и корректная работа с concurrency;
* таймауты, отсутствие бесконечных циклов и очевидных гонок;
* понятные логи и удобство отладки;
* единый стиль кода, хорошие имена, отсутствие дублирования и spaghetti-code.

Под качеством кода в этом ДЗ понимается профессиональный уровень реализации: код легко читать, объяснять и расширять,
а ключевые части системы — quorum, repair, hinted handoff и Merkle sync — выделены и не смешаны друг с другом.
